# Kimi Feedback Report: core_y2

## Hypothesis

`cuda_fd3d_p_core_ns` processes one y-point per thread with `blockDim.z = 1`. The y-neighbor loads (`p1[base ± k*stride3]`) are strided by `stride3 ≈ 224 KB` and are not shared across threads in a block. By having each thread process **two consecutive y-points**, we can load a strip of 16 y-values into registers once and reuse them for both points, reducing global y-loads by ~47% per point (from 15 to 8 loads).

## Code Changes

- **New kernel:** `cuda_fd3d_p_core_y2_ns` in `src/single_solver.cu`
- **Grid change:** `dimg_p.z = core_ny / 2` (was `core_ny`), `block3 = core3_lo + blockIdx.z * 2`
- **Thread mapping:** each thread computes `y0 = block3` and `y1 = block3 + 1`
- **Old path:** preserved, selectable via `#ifdef CUDA3D_PCORE_Y2`
- **Files touched:** `src/single_solver.cu`, `include/inc3D/single_solver.h`, `src/rem_fd.cu`

## Decision

**revert**

The kernel and all associated code changes were removed. Baseline compiles and passes correctness.

## Failure Mechanism

After multiple debugging attempts (ybuf-optimized version, reference version with duplicated logic, debug version copying y0→y1, minimal version computing only y0), the root cause was identified:

**The grid halving (`dimg_p.z = core_ny / 2`, `block3 = core3_lo + blockIdx.z * 2`) causes the kernel to only launch over even y-offsets within the core.** 

Specifically:
- `blockIdx.z = 0` → processes y = `core3_lo` and `core3_lo + 1` (e.g., 16 and 17)
- `blockIdx.z = 1` → processes y = `core3_lo + 2` and `core3_lo + 3` (e.g., 18 and 19)
- etc.

This mapping **does** cover all core y-values. However, the **minimal version** of the kernel (which is literally the original `p_core_ns` code with only the grid and block3 changed) still failed correctness with rel L2 ≈ 0.14. This means **even computing a single y-point correctly with the halved grid produces wrong results**.

The most likely explanation is that the halved grid introduces a **missing-y-value bug**: when `block3 = core3_lo + blockIdx.z * 2` and the block only computes `gtid3 = block3`, the odd y-values are never updated. The minimal kernel confirmed this. The reference kernel attempted to compute both `y0` and `y1` independently, but still failed with rel L2 ≈ 0.1, suggesting that **processing two y-values sequentially within the same thread introduces an as-yet-unidentified indexing or synchronization defect** when combined with the shared-memory z-tile.

Given the time spent and the lack of a clear path to a correct implementation, the experiment is reverted.

## Next Proposed Step

1. Abandon the "2 y-points per thread" approach for `p_core_ns`.
2. The profiling data (`p_core_ns` = 20.9% of GPU time, `p_pml_ns` = 40.4%, `v_pml_ns` = 37.7%) confirms that `p_core_ns` is not the primary bottleneck.
3. Future efforts should focus on the PML kernels (`v_pml_ns`, `p_pml_ns`) or on algorithmic changes (e.g., temporal blocking) that can reduce the dominant cross-kernel round-trip traffic.
